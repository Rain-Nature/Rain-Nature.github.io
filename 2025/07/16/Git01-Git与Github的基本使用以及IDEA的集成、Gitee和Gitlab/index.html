Git01-Git与Github的基本使用以及IDEA的集成、Gitee和Gitlab

Git
Publish Date:   2025-07-16

1. git相关知识扫盲

​ 1.GItLab 是一个基于局域网的代码托管系统，例如公司的项目是闭源项目，不愿意把代码放互联网上

​ 2.Git 是一个本地版本控制工具，本身并不涉及“账户”概念。你设置的user.name和 user.email只是作为每次提交记录的“作者标识”，与远程平台的账户无关。账户管理和权限控制属于远程托管平台（如 GitHub、GitLab），只有在执行推送（push）或拉取（pull）操作时，才需要提供凭证进行身份验证

​ 3.git安装好后，右键有个叫git bash，打开的黑窗口，里面敲的是Linux的命令

2. Git的工作机制

(1)本地操作

​ 工作区：代码存储在本地磁盘的地方，代码可以修改删除。可以将代码添加到暂存区

​ 暂存区：存储在本地仓库目录的 .git/index 文件中，相当于也是存储在磁盘，此时代码也是可以修改删除。可以将代码提交到本地库

​ 本地库：提交到本地库的代码会生成历史版本号，此时代码无法删除。想要删除要把本地库全删了

​

​ 项目被git管理后

你的项目/
├── .git/           ← Git 的核心目录（隐藏）
│   ├── index       ← 暂存区（Stage/Index）就存这里
│   ├── objects/    ← 本地库，所有提交内容、快照
│   ├── refs/       ← 分支、标签
│   └── HEAD        ← 当前分支指针
├── 项目本身文件...

(2)远程管理（代码托管中心）

​ 代码托管中心是基于网络服务器的远程代码仓库，一般我们简单称为远程库。本地库的内容通过push操作可以推送到远程库

  • 局域网：GitLab

  • 互联网：GitHub（外网）、Gitee码云（国内网站）

3. Git的基本使用

Git基本命令览概
命令名称 作用
git config –global user.name <用户名> 设置用户签名（必须且只设一次）
git config –global user.email <邮箱> 设置用户邮箱（必须且只设一次）
git init 初始化本地库
git status 查看本地库状态
git add “文件名” 添加到暂存区
git commit -m <”版本日志信息”> <文件名> 提交到本地库
git reflog 查看历史记录
git reset –hard <版本号> 版本穿梭

(1)从工作区->本地库命令流程

设置用户签名

git config –global user.name <用户名>

git config –global user.email <邮箱>

​ 说明：签名的作用是区分不同操作者身份。用户的签名信息在每一个版本的提交信息中能够看到，以此确认本次提交是谁做的。Git首次安装必须设置一下用户签名，否则无法提交代码

​ 注意：这里设置用户签名和将来登录GitHub（或其他代码托管中心）的账号没有任何关系

初始化本地库

git init

​ 作用：让git获得当前目录的管理权，运行成功后会生成.git文件夹

查看本地库状态

git status

​ 作用：

​ 当文件是红色表面当前文件只存在于工作区

​ 当文件是绿色表面当前文件存入了暂存区

​ 没有文件显示working tree clean，表示当前文件已经提交成功，工作区的文件已经保存在本地库，暂存区无文件

将文件添加暂存区/将文件从暂存区移除

git add <工作区文件名>

git rm –cached <暂存区文件名>

​ 作用：将文件由工作区保存至暂存区/将文件从暂存区移除

暂存区文件提交到本地库

git commit -m <”版本日志信息”> <文件名>

​ 作用：将暂存区文件提交至本地库

查看版本信息（提交记录）

git reflog //查看的是7位版本号（够用于版本切换）

git log //查看的是完整版本号

​ 作用：第一个是查看提交记录，第二个是查看详细记录（有用户名和邮箱）

(2)工作区文件修改后的情况

​ 工作区文件修改后，修改的文件在查看状态时将再次变红，可以添加暂存区，再提交本地库，此时会新增一个新版本进入本地库。再查看版本信息，会发现拥有了两个版本，指针（HEAD->master）会指向最新版本

​ 注意：同一文件不管提交多少个版本，在工作区只会显示1个版本的文件，当查看工作区文件时，此时指针（HEAD）指向本地库哪个版本的文件，工作区就显示那个版本的文件

(3)版本穿梭（切换版本）

git reset –hard <版本号> //这个版本号可以用7位的，不一定要完整版

​ 当使用版本穿梭后，指针（HEAD）会移动到你指定版本号的版本上，如下图所示

​ 使用git完成版本穿梭后，原工作区的文件会变为指针（HEAD）指向的版本的文件，但注意，从暂存区上传文件到本地库后，会更新（HEAD->master）指针指向最新提交

HEAD和master两个指针区别

​ 提交到本地库移动的是master而Head本身就指向master即当前分支，master 是一个普通的分支指针，它指向你本地的最新提交。而版本穿梭是移动HEAD，此时HEAD指向的是要穿梭的版本，工作区显示的内容是HEAD指向版本的内容

​ 关系图

4. 分支（branch）

​ 在版本控制过程中，同时推进多个任务，为每个任务，我们就可以创建每个任务的单独分支。使用分支意味着程序员可以把自己的工作从开发主线上分离开来，开发自己分支的时候，不会影响主线分支的运行。对于初学者而言，分支可以简单理解为副本，一个分支就是 一个单独的副本。（分支底层其实也是指针的引用）

​ 下图是实际开发中的分支图，master为主分支，hot-fix是bug的热修分支，其余两个是功能分支

分支相关的命令览概
命令名称 作用
git branch <分支名> 创建一条分支
git branch -v 查看所有本地分支
git checkout <分支名> 切换分支
git merge <分支名> 把指定的分支合并到当前分支上

(1)查看分支

git branch -v

​ 作用：查看所有本地分支，并附带每个分支上最近一次提交的”版本日志信息”

(2)创建分支

git branch <分支名>

​ 作用：创建一条分支，并将主分支master 的内容复制了一份到新创建的分支上

(3)切换分支

git checkout <分支名>

​ *代表的是当前分支，切换分支从master->hot-fix，如下图所示

​ 切换分支后，对hot-fix里的文件进行了修改后，同样需要先保存到暂存区，然后是提交到本地库，只有提交成功后，这次修改才会成为 hot-fix 分支的历史记录的一部分

​ 注意：切换分支移动的是HEAD指针

(4)合并分支

git merge <你要合并的分支名>

​ 分支合并，需要首先将分支切换到主分支（master）上，然后合并hot-fix分支。合并后，若有相同名称的文件master 分支将包含 hot-fix 分支的改动（master保留了自己原有的内容，同时 引入了 hot-fix 中新增的修改）

(2)GitHub远程仓库的的基本操作

命令览概
命令名称 作用
git remote -v 查看当前所有远程地址别名
git remote add <别名> <远程库地址> 起别名
git push <别名/远程库地址> <本地库的一个分支名>:<远程库的分支名> 推送本地分支上的内容到远程仓库
git clone <远程库地址> 将远程仓库的内容克隆到本地
git pull <别名/远程库地址> <远程分支名>:<本地分支名> 将远程仓库的分支最新内容拉下来与当前本地分支直接合并

​ 作用：推送本地分支上的内容到远程仓库

拉取远程库到本地库

git pull <别名/远程库地址> <远程分支名>:<本地分支名>

​ 作用：将远程仓库的分支最新内容拉下来与当前本地分支直接合并（本地文件自推送后未更改，只是远程库有修改）

​ 注意1：若推送的本地库的分支名和远程库分支名同名，可以简写为：git pull <别名/远程库地址> <远程分支名>

(3)使用Github进行团队内协作（其他人推送我远程库的权限设置）

​ 自己建立的公开远程库，别人只能拉取，不能往上推送代码，要实现团队内的协作，需要给团队成员添加权限

Github团队内协作的步骤-给成员添加权限

​ 1.首先打开GitHub网页

​ 2.输入合作人的GitHub账号名

​ 3.点击添加，此时该成员拥有向代码库推送代码的权限

​ 4.添加成功后，要将链接复制下来，然后将链接发送给令狐冲

​ 5.令狐冲登录GitHub后，将该邀请链接复制到地址栏访问，最后同意邀请即可

​ 6.同意后，该项目在令狐冲的GitHub上也能看见该仓库并访问

(4)Github的跨团队协作

​ 跨团队开发时，不需要在项目中直接赋予权限，而是在提交时审核

Github的跨团队协作的步骤

​ 1.首先打开要协助开发的GitHub仓库，当前登录的是东方不败的账号，看的是岳不群的仓库，点击fork，将岳不群的项目拷贝到东方不败的账号上

​ 2.当前在东方不败的账号下，可以看见该项目了，此时可以对项目进行修改，注意：此时项目修改后不会影响岳不群的项目，可以理解为一份拷贝，当前东方不败对于这个git-demo已经完成修改

​ 3.代码修改完成后，点击Pull request，然后点New pull request，新建推送请求

​ 4.点击Create pull request

​ 5.类似邮件格式写标题、文本，最后点击Create pull request

​ 此时，在岳不群的账号下的Pull Request，可以看到东方不败的请求，点击请求即可查看详细信息

​ 点开后可以看到以下几个圈的地方，第一个圈：当时提交的信件。第二个圈：修改后的文件，点击可以查看。第三个圈是同意将修改的文件与当前远程库合并（需要确认两次）（远程库在东方不败fork之后同一文件未进行更改，否则会导致合并冲突）。第四个圈是实时聊天功能，发送的信息继续像第一个圈那样给东方不败，这个是实时的，不需要刷新网页

​ 合并完成后，此时远程库是可以看到东方不败修改的代码了

(5)Github的ssh免密登录

​ 在之前推送和拉取远程库代码时，使用的是https的远程库地址进行操作的，除了使用https外，还可以使用ssh来进行拉取和推送

ssh免密登录配置步骤
  1. 在终端（Git Bash ）执行(可任意目录)：
ssh-keygen -t rsa -b 4096 -C "你的GitHub邮箱"

​ 注意：执行该命令后需要敲三次回车

  1. 在路径：C:\Users\lausisen\.ssh可以看到公钥文件id_rsa.pub和私钥文件id_rsa

  2. 使用Git Bash 里的cat命令打开公钥文件id_rsa.pub，复制公钥密钥信息（里面全部内容）

  3. 打开 GitHub网站 → 右上头像 → Settings

  4. 左边栏点击 SSH and GPG keys

  5. 点击 New SSH key

  6. Title 随便填，比如 “lxs-SSH-key”

  7. Key里把公钥信息粘贴进去，最后点击Add SSH key即添加成功，如下图所示

ssh免密登录的使用

​ 使用即把远程库地址由https的网址替换为ssh即可

git pull <别名/远程库地址> <远程分支名>:<本地分支名>

​ 其中，第一个是设置远程仓库名，不勾选私有则默认为公共仓库。远程是设置别名。描述可以不填

(3)使用IDEA推送、拉取和克隆代码

推送

​ 项目已经全部提交至本地库后，右键项目->Git->推送

​ IDEA默认是使用https进行推送但速度会偏慢，下图直接点击红色即为https推送。可以修改为使用ssh进行推送，如下图蓝色所示，定义后从仓库复制ssh地址配置，配置好后再点击git-learn选择刚刚配置的ssh再点推送

​ 注意：push是将本地库代码推送到远程库，如果本地库代码跟远程库代码版本不一致， push 的操作是会被拒绝的。也就是说，要想push成功，一定要保证本地库的版本要比远程库的版本高！因此一个成熟的程序员在动手改本地代码之前，一定会先检查下远程库跟本地代码的区别！如果本地的代码版本已经落后，切记要先pull拉取一下远程库的代码，将本地代码更新到最新以后，然后再修改，提交，推送！

拉取

​ 注意：本地代码修改，远程代码也修改，拉取时会有合并冲突，需要手动解决

​ 项目在远程库被别人更新后，需要拉取同步到本地。右键项目->Git->拉取

​ 其中，git-learn是默认的https链接，git-learn2是刚刚配置的ssh链接，后面master是分支。选择好后点击拉取

克隆

​ 打开IDEA的项目界面，点击克隆仓库

​ 其中，URL可以为https或者ssh，配置好后查看目录路径无误，点击克隆
