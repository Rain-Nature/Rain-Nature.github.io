redis02-事务与乐观锁和持久化

redis
Publish Date:   2025-04-29

1. Redis事务和锁机制

​ Redis事务是一个单独的隔离操作：事务中的所有命令都会序列化、按顺序的执行，事务在执行的过程中，不会被其他客户端发送的命令打断

​ 作用：串联多个命令防止别的命令插队

​ 核心理解：Redis中的每条命令都是原子的，但多条命令连起来执行时就不是原子的（Java写的秒杀情况）

(1)事务相关的命令（multi、exec、discard）

​ multi命令：开启事务队列，之后的命令都会进入事务队列中但不执行

​ discard命令：放弃组队，关闭事务队列放弃执行

​ exec命令：执行事务队列中全部的命令

(3)事务冲突的处理（悲观锁、乐观锁）

​ 事务冲突是多个并发事务对相同资源进行修改时发生的“抢夺”，这个时候数据库或缓存就需要决定谁可以成功，谁必须重来或等待

悲观锁

​ 每次别人访问数据时，都认为会修改，所以每次访问数据都加锁，别人必须等我解锁后才能访问

​ 缺点：效率低

乐观锁

​ 在数据访问时给数据加一个版本号的字段，所有人都可以访问数据，但只要有人操作数据，数据库的版本号同步更新，如果操作数据时版本号与数据库的版本号对不上，那么该操作会被回滚

​ 缺点：并发冲突较多时会导致重试开销，且实现上复杂

(4)乐观锁在Redis中的使用

​ 在执行multi之前，先执行watch key1 [key2]…，可以监视一个（多个）key，如果在事务执行之前这个key被其他命令改动，那么事务将被打断，watch等价于Redis中的乐观锁

​ 同时可以使用unwatch取消当前客户端对所有key的监视

样例：redis中使用乐观锁（两客户端模拟并发）

​ 首先的步骤两边同时先将balance监视，然后放入队列中，只是客户端1，先提交

​ 客户端1的watch是自愿监视，可以不需要

(5)Redis 事务的三大特性（总结）

1. 独立的隔离操作

​ 事务中的所有命令会被顺序列队，并在执行时按照顺序执行，事务执行过程中，不会被其他客户端的命令打断

2. 无隔离级别的概念

​ 在执行 EXEC 之前，事务中列队的命令都不会被实际执行

3. 不保证原子性

​ 如果事务中某条命令在执行阶段失败，后续命令仍会继续执行，Redis 不支持事务部分失败时的自动回滚机制

(6)Redis事务秒杀案例

秒杀的基本步骤

​ 1.uid和prodid非空判断

​ 2.连接redis

​ 3.拼接key，用prodid进行拼接库存key（pro：prodid）和秒杀成功的用户key（user：prodid）

​ 4.获取库存如果为null则秒杀还没开始

​ 5.判断用户是否重复秒杀操作

​ 6.判断库存数量小于1秒杀结束

​ 7.秒杀，库存-1，秒杀成功用户加入reids

样例方法：秒杀，需要web环境执行该方法

​ 执行前需要将库存先存入redis：set sk:0101 10

//秒杀案例
public static boolean SecKill(String uId,String propId){
    if (uId == null || propId == null){
        return false;
    }

    Jedis jedis = new Jedis("192.168.199.129", 6379);

    String userKey = "User" + ":" + propId;
    String kcKey = "Kc" + ":" + propId;

    String kc = jedis.get(kcKey);
    if (kc == null){
        System.out.println("秒杀没开始");
        jedis.close();
        return false;
    }

    if (jedis.sismember(userKey, uId)){
        System.out.println("已经参与秒杀");
        jedis.close();
        return false;
    }

    if (Integer.parseInt(kc) < 1){
        System.out.println("秒杀结束");
        jedis.close();
        return false;
    }

    jedis.decr(kcKey);
    jedis.sadd(userKey, uId);
    System.out.println("秒杀成功");

    jedis.close();
    return true;
}
超卖、连接超时的问题

​ 以上代码，由于没有加锁

​ 1.在多并发的环境下会导致超卖的问题

​ 2.在多并发的环境下多请求如果一时间redis无法处理会产生连接超时的问题

超卖、连接超时问题的解决

​ 1.对于连接超时：使用Jedis数据库连接池获取Jedis对象，而不再直接new获取Jedis对象

​ 2.对于超卖：使用乐观锁和事务，锁住库存，代码如下

//秒杀案例（乐观锁优化）
public static boolean SecKill(String uId,String propId){
    if (uId == null || propId == null){
        return false;
    }

    Jedis jedis = new Jedis("192.168.199.129", 6379);

    String userKey = "User" + ":" + propId;
    String kcKey = "Kc" + ":" + propId;

    //开启监视
    jedis.watch(kcKey);

    String kc = jedis.get(kcKey);
    if (kc == null){
        System.out.println("秒杀没开始");
        jedis.close();
        return false;
    }

    if (jedis.sismember(userKey, uId)){
        System.out.println("已经参与秒杀");
        jedis.close();
        return false;
    }

    if (Integer.parseInt(kc) < 1){
        System.out.println("秒杀结束");
        jedis.close();
        return false;
    }

    //使用事务
    Transaction multi = jedis.multi();
    multi.decr(kcKey);
    multi.sadd(userKey,uId);
    List<Object> exec = multi.exec();
    if (exec == null||exec.size() == 0){
        System.out.println("秒杀失败");
        jedis.close();
        return false;
    }

    System.out.println("秒杀成功");
    jedis.close();
    return true;
}
使用乐观锁的问题：库存遗留

​ Redis不能直接使用悲观锁，所以可以使用Lua（一个嵌入式的脚本语言）解决这个问题

​ Lua不支持多线程是单线程的语言，Redis 又是单线程的数据库，这两者结合使得 Lua 脚本在 Redis 中的执行具备了“天然线程安全 + 原子性”的特性，非常适合用于高并发场景中的库存扣减、分布式锁等操作，而出现库存

​ 既然 Redis 是单线程的，那 Java 客户端操作它时，为什么还能出现并发问题？回答：真正的问题是：并发来自客户端，而不是 Redis 本身

​ 编写思路：将Lua脚本写到String中，通过 Java 向 Redis 发送 Lua 脚本，让 Redis 执行这个脚本，再返回结果（一般数字1成功，0失败，2重复）给 Java，Java再将结果判断进一步执行操作

//Java操作Lua+Redis实现秒杀的伪代码，SecKill.java

//具体Lua脚本用时去查，或者那这个去找chatGPT生成
String LuaString = "Lua脚本";
String shal = jedis.scriptLoad(LuaString);
//2是表传两个参数，传入Lua脚本映射关系，KEYS[1] → userid，KEYS[2] → prodid
Object result = jedis.evalsha(shal,2,userid,prodid);
String reString = String.valueOf(result);
if("1".equals(reString)){
	System.out.println("抢购成功");
}else if("0".equals(reString)){
	System.err.println("已抢空");
}

2. Redis持久化

​ Redis提供两种不同形式的持久化方式：RDB和AOF

(1)RDB

​ RDB全称是Redis Database Backup file（Redis数据备份文件），在指定的时间间隔（例如每隔10s）把redis内存中的所有的数据写入磁盘，当Redis实例故障重启后，从磁盘读取快照（dump.rdb文件，默认目录由 dir 配置项指定），恢复数据

​ 缺点：1.最后一次持久化后的数据可能丢失；2.fork()时内存空间需要考虑两倍

RDB底层的执行逻辑

​ 当要生成RDB文件时开始执行

​ 1.主进程调用 fork()，系统生成子进程（通过写时复制技术内存页共享）

​ 2.子进程开始从共享内存页中读数据写 RDB 临时文件，当持久化过程结束后，将该临时文件原子替换为dump.rdb文件

​ 3.主进程继续提供服务（接受写请求）

​ 4.如果主进程改动了某块内存页，系统自动复制给主进程，子进程继续用旧数据

Redis的conf文件对于RDB的配置项

​ 1.dbfilename dump.rdb：设置进行RDB后的文件名称，默认dump.rdb

​ 2.dir ./：设置RDB后文件的路径，./表当前路径（redis-server的启动目录）

​ 3.stop-writes-on-bgsave-error yes：设置当硬盘满时不进行写操作

​ 4.rdbcompression yes：设置是否对持久化文件进行压缩存储，采用LZF算法压缩

​ 5.rdbchecksum yes：设置开启在持久化前检查数据有无损坏

RDB触发备份的方式

​ RDB 每次都会保存“全量数据”，不是只保存变化的“片段”或“增量”

方式 命令或配置 是否阻塞
自动触发 配置文件里的 save 否（后台）
手动触发 命令 BGSAVE 否（后台）
阻塞备份（不推荐） 命令 SAVE 是（会卡主线程）

​ 自动触发的方式：配置为save 20 3，是 “在 N 秒内至少发生 M 次写操作，就触发一次 RDB 持久化，图如下所示

时间 操作 累计写次数（20s 窗口内） 是否触发持久化
0s SET A 1 ❌
1s SET B 2 ❌
2s SET C 3 ✅ 触发 bgsave
3s SET D 1（重新计数） ❌
4s SET E 2 ❌
5s SET F 3 ✅ 可能不触发（间隔太短），不是数据丢失，仅暂时不保存

(2)AOF（Append Only File）

​ 通过将每次写(写入磁盘，包含了修改和删除)命令追加到日志文件中（只许追加到文件不可以修改到文件），redis启动时会读取该文件重新构建数据实现数据恢复

AOF的使用（默认没开启）

​ 1.在redis.conf中将appendonly no 改为 yes

​ 2.appendfilename “appendonly.aof”：设置AOP文件名，AOP的生成路径与RDB同步

AOF的特点

​ 当redis目录同时有AOF和RDB文件时，默认读取AOF文件里的信息

AOF文件的异常恢复

​ 如果遇到AOF文件损坏，如打开vim在AOF文件中加了不明字段，此时AOF文件就已经损坏，可以通过/usr/local/bin/redis-check-aof–fix appendonly.aof进行恢复

AOF的其他配置：同步频率设置

​ 1.appendfsync always：始终同步，每次Redis的写入都会立即记入日志，性能较差但数据完整性好

​ 2.appendfsync everysec：每秒同步，每秒记入日志一次，如果宕机，本秒的数据可能丢失

​ 3.appendfsync no：redis不主动进行同步，把同步时机交给操作系统

rewrite（重写压缩）

​ 把相同类型的指令，重写压缩为一条指令，只关注结果一致，过程与RDB保存类似，都使用了fork()，当文件大小大于64Mb的两倍时触发重写

(3)RDB和AOF的对比

​ 官方推荐两个都使用，如果对数据不敏感可以单独用RDB，AOF不建议单独使用，如果只做纯内存缓存可以都不使用

两者的使用逻辑
  • AOF 保证数据不丢，RDB 保证恢复效率快
  • 重启时优先用 AOF 恢复，如果 AOF 不可用才回退 RDB
两者的保存数据的逻辑
  • Redis 正常运行时，AOF 追加写操作的日志
  • RDB 只有在触发 save 或 bgsave 时，才将快照当前内存状态写入 .rdb 文件；
  • 它们各写各的，互不感知，没有“一个完成了就同步另一个”的逻辑，但rdb可以人工同步，aof仅自动触发不能人工同步
两者的对比表
对比项 RDB 快照 AOF 日志
核心作用 完整备份（定时快照） 数据安全（持续写操作记录）
写入频率 低（定时/手动触发） 高（每次写操作都记录）
恢复速度 快，直接恢复内存状态 慢，要逐条执行命令
文件体积 通常较小 通常较大，但可压缩重写
数据精度 可能丢失最近几秒的数据 最多丢 1 秒数据（everysec 配置）
适合场景 做冷备、定期快照、快速大数据恢复 做热备、防止数据丢失，容灾能力更强
