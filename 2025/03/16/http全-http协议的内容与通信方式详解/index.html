http全-http协议的内容与通信方式详解

Web开发协议
Publish Date:   2025-03-16

1. HTTP协议

​ HTTP协议即按一定规则，向服务器要数据或发送数据，而服务器按一定规则回应数据

​ HTTP协议主要约定了请求 和响应

2. HTTP的连接

​ 0：原始状态，客户端和服务器未连接

​ 1：客户端向服务器发送请求

​ 2：服务器沿着连接响应信息

​ 3：客户端将收到的内容显示，过了几秒后双方断开连接

3. http请求信息和响应信息的格式

(1)使用telnet模拟HTTP请求（GET）

请求报文：

​ (1)请求行

​ (2)请求头信息

​ (3)空行

​ (4)请求主体信息(可以没有)

​ 注意：头信息结束后要空一行，如果请求有主体信息需要配置请求的Content-Length，告知主体信息长度，然后空一行写主体信息

请求行详解

​ 请求行包括

​ 1.请求方法(GET)：还有（GET|POST|PUT|DELETE|TRACE|OPTIONS）

​ 2.请求路径(/0606/01.php) ：是URL的一部分

​ 3.所用协议(HTTP/1.1)：HTTP协议1.0，0.9基本没人用了

​ 注意：这些请求方法虽然HTTP协议规定，但WEB SERVER未必允许或支持这些方法

响应报文：

​ (1)响应行(协议 状态码 状态文字)

​ (2)响应头信息(格式为 key：value)（可能有多行）

​ (3)空行

​ (4)主体信息(可能没有)：响应返回的结果一般是html

(2)发送请求时同时要发送数据给服务器(POST)

​ 一般在输入完Host：localhost后，就会向服务器发送请求，如果要发送数据的话，需要对请求做如下修改，且集中粘贴过去

POST /0606/02.php HTTP/1.1
Host:localhost
Content-type:application/x-www-form-urlencoded   //告诉服务器会发送POST的数据包，没有此行服务器无法正确接受数据
Content-length:23  //告诉服务器请求主体的长度

​ 运行图示

4. 状态码，状态文字

​ 状态码是用来反应服务器响应情况的，状态文字是用来描述状态码，便于人观察的

(1)状态码一共有5个系列，不用一个一个背

状态码 定义 说明
1XX 信息 接收到请求，继续处理
2XX 成功 操作成功地收到，理解和接受
3XX 重定向 为了完成请求，必须采取进一步措施
4XX 客户端错误 请求的语法有错误或不能完全被满足
5XX 服务端错误 服务器无法完成明显有效的请求

​ 1XX：是给HTTP留着以后扩展用的

常见的状态码(背)：

  • 200 - 服务器成功返回网页（请求成功），在Spring中被用作数据成功返回

  • 301/2 - 永久/临时重定向(服务器内部跳转)

    重定向的响应中有(Location)的属性来定位重定向的网页是谁

  • 304 - 未修改（Not Modified），取的是缓存，多次加载同一界面会出现

    通过客户端向服务器发送时间锉信息和标签(第一次请求获得的响应)看时间是否过期，标签是否修改

  • 307 - 重定向中保护原有的请求数据

    发送POST数据到05.php，05.php重定向到06.php显示数据，如果用301/2数据会丢失，但307可以保护数据，确保数据在06.php正常显示

  • 404 - 请求的网页不存在

  • 503 - 服务器暂时不可用

  • 500 - 服务器内部错误

5. HTTP和TCP（面试题）

​ HTTP是半双工的，TCP是全双工的，浏览器若需要打游戏需要使用全新的WebSocket来进行连接，第一次三次握手时，浏览器发送的是普通的HTTP请求，若要建立WebSocket需要在HTTP请求中带特殊的head头

​ HTTP协议是基于TCP实现的

​ 浏览器的核心功能之一就是发起 HTTP 请求并解析响应

​ 以打开百度搜索“刘大爷”为例

​ 注意：请求体信息是在与Request Headers同级的叫Form Data中

7. 用户身份认证和状态保持

​ Cookie、Session 和 Token 都是用于管理用户身份认证和状态保持

方案 存储位置 状态管理 适用场景 优缺点
Cookie 浏览器（客户端的一种） 有状态 记录小型用户信息 易受攻击，大小受限
Session 服务器 有状态 适合中小型系统 服务器内存占用大
Token (JWT) 客户端（任何能发http的应用） 无状态 前后端分离、微服务 无法主动销毁，易泄露

​ 注意：有状态和无状态，主要指的是服务器是否需要记住客户端的信息，HTTP请求本身是无状态的

(1)Cookie

​ Cookie 是实现每次HTTP请求都自动带数据给服务器的技术，它也是存储在浏览器端的一小段数据。等于有两个功能，一个是存储，另一个的发送HTTP自动携带这些数据。图如下所示

特点
  • 存储在浏览器中，打开浏览器可直接查看
  • 安全性较低，容易被劫持
适用场景
  • 存储简单的状态信息（如用户偏好）

(2)Session

​ Session 存储在服务器端。用户第一次访问服务器时，携带账号密码进行登录，服务器验证通过后，会生成一个 SessionID（随机字符串）和 Session 数据，并设定有效期，然后通过 Cookie 将 SessionID 发送给客户端。客户端会自动存储这个 Cookie，之后的请求无需再输入密码，而是用Cookie携带 SessionID发送。服务器收到请求后，通过 SessionID 查找对应的 Session 数据，确定用户身份。图如下所示

特点
  • 存储在服务器上
  • 需要 Cookie 或 URL 传递 SessionID
  • 生命周期：默认情况下，Session 在用户关闭浏览器后会失效（但可以配置 Session Timeout）
适用场景
  • 用户登录状态管理（适用于小规模应用，不适用微服务）
  • 存储临时的用户信息（如购物车）

(3)Token（JWT）

​ 多服务器的条件下Session已经不再适用，当客户端发送密码后，服务器端生成JWT（服务器不保存），只保存JWT的密文，然后将JWT发送给客户端，用Cookie存储，再次请求服务器时，只需要发送JWT，与Session类似。图如下所示

特点
  • 不需要服务器存储状态（无状态）
  • 可以跨域、适用于微服务架构
  • 一般存储在 LocalStorage 或 Cookie 中
适用场景
  • 前后端分离的身份认证
  • 移动端登录
  • 微服务架构

(4)开发中的选择

​ 1.小型项目（单体应用）
​ → Session + Cookie（简单好用）

​ 2.前后端分离
​ → Token（JWT）+ Authorization: Bearer token

​ 3.需要支持分布式（多个服务器）
​ → Redis 共享 Session 或 Token 方案
